# 产品需求文档："流式知识河"AI学习助手

| 文档版本 | 1.0 |
| :--- | :--- |
| **产品名称** | 流式知识河 (Streaming Knowledge River) |
| **目标用户** | 理工科大学生（主要使用场景：高等数学、线性代数、大学物理学习） |
| **核心痛点** | 1. 阅读教材或讲义时存在概念理解障碍，读不懂、记不住。<br>2. 学后知识呈碎片化，无法有效关联新旧知识，缺乏体系。<br>3. 传统AI对话为线性流式输出，难以把握整体知识结构，重点内容梳理困难。<br>4. 缺乏对自己"不懂"和"犯错"知识点的有效沉淀与回顾机制。 |
| **产品愿景** | 打造一个以**对话流驱动、注重知识点自然关联与沉淀**的本地化AI学习伴侣，让学习者在与AI的自由问答中，自然形成结构化的知识脉络，并专属保存认知突破时刻的"知识财富"。 |

---

## 一、用户角色与核心用户故事

### 1.1 用户角色
- **姓名**：李理
- **身份**：某高校理工科大二学生，正在学习《线性代数》和《大学物理》。
- **行为特征**：
    - 习惯于看PDF教材和讲义自学。
    - 遇到不懂的概念习惯直接提问，但不喜欢被灌输大量预设结构。
    - 重视自己的思考轨迹和纠错过程，认为"搞懂自己错在哪"比"记住正确答案"更重要。
    - 对庞大、静态的思维导图有畏难情绪。

### 1.2 核心用户故事
1.  **作为李理，我希望**上传PDF教材章节后，AI能直接给我一个清晰详细的解答作为起点，而不是一个复杂的知识树，这样我可以**立刻开始学习，不被干扰**。
2.  **作为李理，我希望**在我针对具体问题提问时，系统能**实时提示我当前问题在整个知识体系中的位置**，让我始终有方向感，不会迷失在细节里。
3.  **作为李理，我希望**对话结束后，系统能**自动生成我本次的"探索路径图"**，清晰展示我从哪里出发、重点攻克了哪些知识点，这样我能直观回顾自己的思考过程。
4.  **作为李理，我希望**在学习中，可以随时将那些"我原本不懂或理解错了"的关键认知，**一键保存为一张精美、格式化的知识卡片**，并能添加自己的注解。这些卡片是我真正的知识财富，必须保留下来。
5.  **作为李理，我希望**当我学习新章节或遇到类似问题时，AI能**主动提示我："这与你之前保存的XX知识卡有关联"**，帮助我把新旧知识串联起来，实现真正的积累。

## 二、功能性需求

系统核心流程围绕"单次学习会话"展开，并在多次会话间建立知识关联。

### 2.1 模块一：学习会话初始化

| 功能ID | 功能名称 | 详细描述 |
| :--- | :--- | :--- |
| **F-01** | **PDF文档上传与解析** | 用户可上传一个或多个PDF文件（支持高等数学、线性代数、物理等包含公式的文档）。系统后台完成文本和公式的提取与预处理。 |
| **F-02** | **初始解答生成** | 上传完成后，AI直接生成一份针对该PDF核心内容的**详细解答**。该解答应以**自然语言为主**，风格清晰、直接，避免复杂的图表结构，作为用户提问的起点。 |

### 2.2 模块二：流式问答与知识锚定

这是产品核心交互区，强调在对话中建立结构感。

| 功能ID | 功能名称 | 详细描述 |
| :--- | :--- | :--- |
| **F-03** | **实时知识位置显示** | 在AI回答框的上方或侧边，通过**面包屑导航**（如：`线性代数 > 第三章 矩阵 > 矩阵的秩 > 性质`）实时显示当前讨论主题在文档中的逻辑位置。用户可点击面包屑导航的任意层级，查看该层级的知识概述。 |
| **F-04** | **上下文感知问答** | 用户可在初始解答基础上，以多轮对话形式自由提问。AI需结合上传的PDF内容与当前对话历史，给出准确、有上下文的回答。 |
| **F-05** | **学习足迹自动打点** | 系统在后台**自动记录**用户的每一次有效提问（即触发AI回答的提问）。该记录作为后续生成"学习足迹图"的原始数据点。 |
| **F-06** | **一键摘录与批注** | 用户可选中AI回答中的任意文本段落，点击"**摘录为知识卡**"。弹出窗口允许用户**添加自己的批注**（如："我错在没考虑前提条件"），并保存。 |

### 2.3 模块三：学习成果与产出

解决用户"学完即走，无沉淀"的问题。

| 功能ID | 功能名称 | 详细描述 |
| :--- | :--- | :--- |
| **F-07** | **学习足迹图生成** | 一次学习会话结束后，系统根据记录的学习足迹（F-05），生成一张**可视化的"探索路径图"**。该图应为**放射状或线性动线图**，起点是初始解答（F-02），各节点为用户提问的主题词，连线表示提问顺序。 |
| **F-08** | **知识卡片管理** | 提供一个独立的"知识卡库"视图。所有通过摘录（F-06）生成的知识卡，以**瀑布流或网格形式**展示。每张卡片包含：摘录原文、用户批注、来源文档、创建时间。支持按课程或关键词检索。 |

### 2.4 模块四：长期知识关联与回顾

让工具从"一次性用品"变为"长期学习伴侣"。

| 功能ID | 功能名称 | 详细描述 |
| :--- | :--- | :--- |
| **F-09** | **跨会话智能关联提示** | 当用户在新会话中提问时，系统需在后台检索**历史知识卡库**和**历史会话足迹**。若发现关联，则在AI回答下方给出**温和的文字提示**："这个概念与你之前在《XXX》中保存的'YYY'知识卡有关，需要回顾吗？"并提供点击跳转查看的入口。 |
| **F-10** | **全局智能检索** | 提供一个全局搜索框，用户可输入关键词。系统能够检索：1）所有上传过的PDF文件名及元数据；2）所有历史对话记录的文本内容；3）所有知识卡片的标题、原文和批注。搜索结果需高亮关键词并分类展示。 |

## 三、非功能性需求

| 类别 | 详细描述 |
| :--- | :--- |
| **本地化与隐私** | 所有数据处理、模型推理、数据存储均须在**用户本地设备**完成。不上传任何用户文件或对话数据至云端。 |
| **性能** | PDF解析和初始解答生成应在用户可接受时间内完成（如：20页以内PDF，处理时间 < 30秒）。问答响应应保持流畅，无明显卡顿。 |
| **数据持久化** | 系统需可靠保存所有用户数据，包括：上传文档的索引、完整对话历史、生成的学习足迹图、知识卡片库。支持数据导出与备份。 |
| **易用性** | 界面设计应**简洁、克制**，突出对话流与知识卡片。避免过度复杂的专业术语，交互逻辑需符合学生直觉。 |
| **可扩展性** | 系统架构应支持未来增加新的课程领域，或引入新的AI模型，而无需对核心流程做大改动。 |

## 四、交互设计核心原则

1.  **流式优先，摒弃静态树**：知识的体系感应在对话流动中通过"面包屑导航"和"足迹图"自然呈现，而非一开始就展示一个令人生畏的思维导图。
2.  **聚焦"人味"关联**：知识关联不由僵硬的图谱驱动，而是由AI用**自然语言提示**和**回顾入口**来实现，像一位懂你的学伴在提醒你。
3.  **沉淀"认知黄金"**：产品的核心价值之一是帮助用户捕捉并美化"顿悟时刻"和"纠错瞬间"。知识卡片的生成和浏览体验必须是**优雅且令人愉悦**的。
4.  **本地优先，数据自主**：所有设计决策均需遵循数据不离本地的基本原则，给予用户完全的数据掌控感。

## 五、验收标准

1.  能够成功上传高等数学PDF，并看到AI生成的初始详细解答。
2.  能够在问答时，看到AI回答上方准确显示当前话题的面包屑导航位置。
3.  能够手动摘录AI回答中的文本，添加批注，并成功保存为一张知识卡片。
4.  完成一次包含至少3次提问的会话后，能够看到一张可视化的学习足迹图。
5.  在新会话中提问与旧知识相关的问题时，能看到AI给出的跨会话关联提示。
6.  所有用户数据（索引、对话、卡片）在关闭并重启应用后依然存在。

## 六、技术设计决策（MVP）

> 本节为 MVP 实现的技术约束与设计决议，已确认结论不再作为待讨论项。

### 6.1 已确认决议

1. 中栏首版使用"结构化正文视图"，不做高保真 PDF 复刻。
2. 左右侧栏首版使用"点击展开/固定"，不做悬停自动展开。
3. 核心三栏区采用 `Streamlit 自定义组件` 实现。
4. 知识河归档用 `sessions.status = archived` 表达，不新增冗余归档总表。
5. 页面坚持"按阶段显隐"，不全量展示信息。
6. LaTeX 公式渲染为核心能力，必须进入 MVP。

### 6.2 主页面核心原则

#### 专注优先
1. 任意时刻只展示当前任务所需信息。
2. 用户视线焦点始终在"中栏阅读"或"右栏提问"二者之一。
3. 非关键信息收纳到左栏，不抢占中栏空间。

#### 渐进显隐
1. 初始化前只展示上传入口。
2. 解析完成后，中栏上传入口消失，不再占位。
3. 与提问相关内容只在右栏出现，不在中栏重复。

#### 本地优先
1. 所有解析、检索、问答、存储在本地执行。
2. 所有状态可恢复，避免学习过程丢失。

### 6.3 信息架构

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部栏：课程/会话状态/保存状态/汇入知识河                     │
├──────────────┬───────────────────────────────┬──────────────┤
│ 左栏         │ 中栏                           │ 右栏         │
│ 会话档案     │ 结构化正文阅读 + 面包屑         │ 提问与回答    │
│ 历史恢复     │ （上传区仅初始化可见）           │ （选中文本触发）│
├──────────────┴───────────────────────────────┴──────────────┤
│ 底部：学习足迹图（按需展开，不默认占屏）                      │
└──────────────────────────────────────────────────────────────┘
```

### 6.4 三栏职责与状态

#### 左栏（知识档案）
- 内容：文档信息、会话摘要、历史会话列表、`汇入知识河` 按钮
- 规则：默认折叠，可点击展开，可固定

#### 中栏（阅读核心区）
- 内容：面包屑导航、结构化正文、阅读内联提示
- 规则：上传区只在初始化阶段显示，解析后移除

#### 右栏（提问与回答区）
- 触发：选中文本后自动展开
- 内容：选区预览、快捷动作、自由提问框、流式回答

### 6.5 页面生命周期

| 阶段 | 中栏 | 左栏 | 右栏 |
|------|------|------|------|
| A：初始化 | 上传控件 | 空状态 | 关闭 |
| B：解析中 | 进度提示 | 空状态 | 关闭 |
| C：解析完成 | 面包屑+正文 | 文档元信息 | 默认折叠 |
| D：问答与沉淀 | 正文 | 会话档案 | 展开 |

### 6.6 LaTeX 公式渲染

- 渲染目标：中栏正文与右栏回答中的公式可读、可复制、可选中
- 渲染策略：后端输出标准数学标记（`$...$`、`$$...$$`），组件内使用 MathJax/KaTeX
- 流式策略：文本块完成后再触发局部公式渲染，避免每个 token 重排

### 6.7 数据模型

保留 5 张核心表：
1. `documents` - 文档元数据
2. `sessions` - 会话生命周期（`status` 仅 `draft/archived`）
3. `messages` - 问答记录
4. `knowledge_cards` - 摘录批注
5. `footprints` - 学习足迹

约束：不新增"归档总表"，不做过度 JSON 大字段堆叠，优先可检索、可恢复、可维护。

### 6.8 工程组织原则

#### 模块划分
- `app.py`：页面入口与路由
- `services/`：业务逻辑层（session_service、qa_service、document_service 等）
- `repositories/database.py`：SQLite 读写
- `models/schemas.py`：Pydantic 数据模型

#### 反模式（明确禁止）
1. 为每个小按钮拆独立模块，导致追踪困难。
2. 把业务逻辑塞进前端脚本，后端失去可测性。
3. 为"理论完美"引入过多抽象层和接口层。

#### 质量底线
1. 核心流程函数保持短小且可读。
2. 单文件可接受，但必须按职责分段并有清晰命名。
3. 每个核心模块至少有最小可运行验证。

### 6.9 MVP验收清单

1. 初始化时，中栏有上传区；解析成功后中栏上传区消失，文档信息迁移到左栏。
2. 中栏稳定显示结构化正文与面包屑，且正文可选中。
3. 选中文本后右栏展开，能进行快捷提问与流式回答。
4. AI 回答可摘录成知识卡并附批注。
5. 会话自动保存为 draft，点击"汇入知识河"后状态变为 archived。
6. 学习足迹可生成并可查看。
7. 中栏与右栏中的 LaTeX 公式均可正确渲染。
8. 页面不全量堆叠信息，交互遵循"当前任务优先显示"。
